# TUI: Terminal UI for varuna using zigzag and zio

## What was done

Added `varuna-tui`, a new rtorrent-style terminal user interface binary that
communicates with the varuna daemon over its qBittorrent-compatible WebAPI.

### Architecture

- **zigzag** drives the TUI rendering using the Elm Architecture pattern
  (Model-Update-View). It handles terminal I/O, keyboard input, screen refresh,
  and widget styling.
- **zio** runtime is initialized for async I/O infrastructure. The HTTP client
  uses `std.net` TCP connections from background threads (via zigzag's
  `AsyncRunner`), since the polling runs off the main thread.
- **Clean separation**: `src/tui/api.zig` (HTTP client + JSON parsing),
  `src/tui/views.zig` (all rendering components), `src/tui/main.zig` (event
  loop, model, key handling).

### Features implemented

**Main view:**
- Torrent list with columns: status icon, name, size, progress bar + %, download
  speed, upload speed, seed count, peer count
- Scroll support with j/k and arrow keys, Home/End
- Selected row highlighting with background color
- Empty state message

**Status bar:**
- Global download/upload speeds, torrent count, DHT node count
- Connection status indicator

**Key bindings bar:**
- Context-sensitive hints that change per view mode

**Modal dialogs:**
- Add torrent (file path or magnet URI, Tab to switch mode)
- Confirm delete (with toggle for deleting files from disk)
- Preferences view (read-only display of daemon settings)

**Detail view:**
- Torrent header (name, status, progress, speeds, seeds/peers, size, ratio, path)
- Three tabs switchable with Tab: Files, Trackers, Info
- Files tab shows index, name, size, completion %, priority
- Trackers tab shows URL and status
- Info tab shows hash, save path, pieces, connections, dates, category, tags

**Navigation:**
- q/Esc: quit or go back
- j/k, Up/Down: navigate torrent list
- Enter: open torrent details
- a: add torrent dialog
- d/Delete: delete torrent dialog
- p: pause/resume toggle
- P: preferences view
- /: filter mode
- Tab: switch detail tabs

**HTTP API client:**
- Raw HTTP/1.1 client using `std.net.tcpConnectToAddress`
- Cookie-based session (SID) support for authentication
- JSON response parsing via `std.json`
- Endpoints: torrents/info, transfer/info, torrents/properties, torrents/files,
  torrents/trackers, app/preferences, torrents/add, torrents/delete,
  torrents/pause, torrents/resume, auth/login

**CLI:**
- `--host HOST` and `--port PORT` options (default: 127.0.0.1:8080)
- `--help` flag with full usage documentation

### Build integration

- New `varuna-tui` executable target in `build.zig`
- New `zig build tui` run step
- zigzag and zio added to `build.zig.zon` dependencies
- Binary does NOT depend on the varuna daemon's internal modules

## What was learned

- Zig 0.15 changed `std.ArrayList(T)` to be the unmanaged version (no allocator
  stored). Use `.empty` initialization and pass allocator to each method call, or
  use `std.fmt.allocPrint` for string building which is more idiomatic anyway.
- zigzag's `Style.width()` takes `u16`, not `usize`. Need explicit cast with
  `@intCast(@min(val, maxInt(u16)))`.
- zigzag's `KeyEvent.key.char` is `u21` (Unicode codepoint). Need `@intCast(c)`
  when storing into a `u8` buffer, with a `c < 128` guard for ASCII.
- Zig 0.15 `std.fs.File.Writer` uses `.interface.writeAll()` not `.writeAll()`.
- zigzag's `AsyncRunner` + `everyMs()` tick provides a clean periodic polling
  mechanism for API updates without needing a separate event loop.
- zio's fiber/coroutine model requires a `zio.Runtime` context to run. Since
  `AsyncRunner` tasks run on plain OS threads, using `std.net` for HTTP is more
  practical than zio's `net.Stream` for background polling.

## Remaining issues / follow-up work

- **Action dispatch**: Pause/resume, add, and delete operations log the intent
  but don't yet spawn background HTTP tasks. Need to wire `AsyncRunner.spawn`
  to API client calls with proper error reporting.
- **Detail data fetching**: Entering detail view doesn't yet trigger background
  fetches for files/trackers/properties. Same wiring pattern needed.
- **Filter**: Filter mode captures input but doesn't yet filter the torrent list
  by name substring.
- **Reconnection**: The polling loop handles disconnection gracefully (shows
  overlay) but could add exponential backoff.
- **Memory**: The arena allocator in zigzag's view function handles per-frame
  allocations, but the API result data from `pollDaemon` uses
  `page_allocator` and is never freed (leaks until process exit). Should use
  a ring buffer or double-buffered arena for API results.
