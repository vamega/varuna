# API tests: external-review T2 + T3 + T4

External-review verification flagged three test-coverage gaps:

- **T2** — private-torrent → DHT gating was claimed but never asserted.
- **T3** — every API test bypassed the real HTTP routing layer; some
  used a dummy echo handler instead of the real `ApiHandler`.
- **T4** — API tests skewed heavily negative (missing-param, unknown-
  hash); happy-path coverage for mutating endpoints was thin.

This commit set closes all three.

## What changed

### T2 — Private-torrent DHT gate (one-place enforcement + tests)

**Refactor**: the privacy gate (BEP 27 — private torrents must not
leak to the DHT) was previously inline `if (!self.is_private)` in
three call sites:

- `src/main.zig:294` (initial DHT register-for-peers)
- `src/daemon/torrent_session.zig:688` (post-event-loop force-requery)
- `src/daemon/torrent_session.zig:1457` (seed-mode announce)

That left three places to drift if a future refactor missed one. I
collapsed all three into helper methods on `TorrentSession`:

```zig
pub fn dhtRegisterPeers(self, engine) bool;   // wraps requestPeers
pub fn dhtForceRequery(self, engine) bool;    // wraps forceRequery
pub fn dhtAnnouncePeer(self, engine) bool;    // wraps announcePeer
```

Each gates internally on `self.is_private` and returns `true` only
when the call was actually issued. `dhtV2HashTruncated` was promoted
from `fn` to `pub fn` because the helpers (and the new tests) need
it. Behaviour at the call sites is identical — same v1+v2 hash
fan-out, same error-swallowing for `announcePeer`.

**Tests** (`tests/private_torrent_dht_test.zig`, 9 tests):

- Private metainfo round-trips: `is_private` true for `private:1`,
  false for `private:0`.
- For each of the 3 helpers: private torrent is a no-op; public
  torrent reaches the engine. Engine state asserted via
  `pending_search_count`, `pending_search_done[i]`, `listen_port`,
  and `active_lookups`.
- Hybrid v2 coverage: private hybrid stays gated even when both v1
  and the truncated v2 hash would otherwise register; public hybrid
  registers BOTH hashes.

The strategy is "real DhtEngine + real TorrentSession parsed from a
private/public metainfo blob"; no full event loop is needed because
the helpers talk directly to the engine. Layer-1 unit tests per
`STYLE.md`'s Layered Testing Strategy.

**No production bug surfaced** — the gate was already in all three
sites. The tests pin the contract going forward.

### T3 — Real ApiServer routing tests

`tests/api_endpoints_test.zig` constructed `ApiHandler` directly on
the stack, skipping the routing layer entirely.
`tests/rpc_server_stress_test.zig` used `echoHandler` so its
lifecycle assertions wouldn't be tangled with handler logic. Neither
test surface caught regressions in the parts between the kernel and
the handler.

Added 6 new tests in `tests/rpc_server_stress_test.zig` that wire a
real `ApiHandler` (with `SessionManager`, sync_state, peer_sync_state)
onto a real `ApiServer` running on `RealIO`, then fire HTTP requests
through `posix.connect` + `posix.write` + `posix.read`:

- POST /auth/login: reachable without auth, returns 200 + Set-Cookie.
- GET /app/version without auth: returns 403 (auth gate works).
- GET /app/version, /app/buildInfo, /app/defaultSavePath with auth:
  all 200, expected bodies (covers 3 routes for the price of one).
- GET /torrents/info: returns `[]` from the real handler (proves the
  request reached `handleTorrentsInfo`, not the catch-all 404).
- POST /torrents/createCategory + GET /torrents/categories: full
  round-trip through routing → handler → SessionManager mutation.
- GET /api/v2/this/is/not/a/route: returns the handler's JSON 404,
  not the server's text 404 (confirms `setHandler` wiring).

The handler is bound via a file-scope global (`routing_handler_global`)
because `setHandler` takes a function pointer — same pattern
`src/main.zig` uses for `globalApiHandler`. Tests are sequential, so
the global is safe.

### T4 — Happy-path API coverage (4 files, 25 tests)

`tests/api_endpoints_test.zig` covers only the validation paths.
This adds positive paths in 4 endpoint-family files:

**`tests/api_categories_test.zig`** (6 tests):

- `createCategory` → list shows it; `setCategory` on a torrent
  observable through `session.category`.
- `editCategory` updates `savePath`.
- `removeCategories` clears the assigned torrent's category back to
  `null` (production walks the sessions map and frees in-place).
- `setCategory` with empty string clears.
- `createTags` + `addTags` + `removeTags` round-trip; `deleteTags`
  also prunes from each torrent's tag list.
- `addTags` is idempotent.

**`tests/api_share_limits_test.zig`** (4 tests):

- Round-trip: `setShareLimits` → `torrents/properties` JSON contains
  `"ratio_limit":2.5000`, `"seeding_time_limit":4320`. Pins the
  user-visible contract (the output formatter prints `ratio_limit` /
  `seeding_time_limit`, not the internal field names).
- `-1` (no-limit sentinel) and `-2` (use-global default) both stick.
- Pipe-separated multi-hash applies to all hashes.

**`tests/api_tracker_edits_test.zig`** (7 tests):

- `addTrackers` populates `tracker_overrides.added`.
- `addTrackers` with `%0A` separator parses multiple URLs.
- `editTracker` on a metainfo URL records an `edit` override.
- `editTracker` on a previously-added URL replaces in-place (no
  edit override).
- `removeTrackers` on an added URL drops it from `added`.
- `removeTrackers` on a metainfo URL records a `remove` override.
- `addTrackers` is idempotent (de-dupes against existing added URLs).

The tests don't require a started session because the mutators only
touch the in-memory `TrackerOverrides` struct.

**`tests/api_sync_export_test.zig`** (8 tests):

- `rename` updates `session.name` in place; rejects empty name.
- `export` returns the exact metainfo bytes the session was created
  from (byte-for-byte slice compare).
- `addPeers` reaches the handler and returns 200 (peer queueing
  itself requires the event loop, so this test pins handler reach
  but not the queueing — see "Constraints" below).
- `sync/maindata?rid=0` first call: full_update=true, rid=1, torrent
  hash present in body.
- Second call with `rid=1`: full_update=false, rid=2, unchanged
  torrent NOT present in delta.
- After applying a tag, the next delta must include the torrent
  again (stats hash changed).
- Removing a session surfaces its hash in `torrents_removed`.

## Constraints / surprises

1. **`renameTorrent` doesn't persist across daemon restart.** The
   handler updates `session.name` in memory but doesn't call any
   resume-DB save method. The brief asked for "rename persistence
   across torrent stop+start"; the production code doesn't actually
   persist this today. Test asserts the in-memory contract; the
   persistence gap is a real follow-up (not surfaced as a test
   failure because no such test existed before this work). Tracked
   as out-of-scope here; csprng-engineer / docs follow-up should
   decide whether to extend `state_db.zig` with a per-torrent name
   override row or accept that "rename is display-only and resets
   on restart."

2. **`addPeers` happy-path requires the event loop.**
   `SessionManager.addManualPeers` short-circuits when
   `shared_event_loop` is null. Test pins handler reach + 200, not
   the queueing itself. Full happy-path needs an `EventLoopOf(SimIO)`
   harness, which is heavier than the rest of T4 and was deferred.

3. **Bencode length prefixes are silent failures.** The first cut
   of `tests/api_tracker_edits_test.zig` declared a tracker URL with
   the wrong length prefix; the bencode parser silently mis-tokenises
   and `findInfoBytes` returns `MissingInfoDictionary`. Fixed by
   length-counting carefully and adding a comment noting the trap.
   No production bug — purely a test-fixture issue.

4. **Zig 0.15.2's "var never mutated" is strict.** The test pattern
   `var sess = try TorrentSession.create(...); defer sess.deinit();`
   fails the check because `defer .deinit()` doesn't count as
   mutation. Fixed by heap-allocating sessions and operating through
   pointers — same pattern `SessionManager.addTorrent` uses.

5. **`getSessionTrackers` requires a started session.** Without an
   event loop, `session.session` is null and the call returns
   `error.TorrentNotReady` (409). The tests that need to verify
   tracker mutations therefore inspect `session.tracker_overrides`
   directly instead of going through `torrents/trackers`. The new
   stress-test `T3` set still exercises one full handler round-trip
   for the routing-layer assertions.
