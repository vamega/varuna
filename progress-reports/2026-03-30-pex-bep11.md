# BEP 11: Peer Exchange (PEX) Implementation

## What was done

Implemented BEP 11 Peer Exchange, building on the existing BEP 10 extension protocol infrastructure. PEX allows peers to exchange information about other peers in the swarm, reducing dependence on tracker announces for peer discovery.

### New files
- `src/net/pex.zig` -- PEX message parsing, generation, state tracking, and tests

### Modified files
- `src/io/protocol.zig` -- incoming PEX message handling, peer connection from PEX, PEX message sending
- `src/io/peer_policy.zig` -- periodic PEX tick (checkPex) called every event loop cycle
- `src/io/event_loop.zig` -- PEX state in Peer struct and TorrentContext, cleanup on disconnect/deinit
- `src/net/root.zig` -- export pex module
- `STATUS.md` -- updated done/next sections

### Key design decisions

1. **Delta encoding**: Each peer tracks what it has previously told each connected peer (PexState.sent_peers). On each PEX tick, we compute the diff between the torrent's connected peer set and what this peer knows, sending only added/dropped deltas. This matches BEP 11 requirements and minimizes bandwidth.

2. **Per-torrent connected peer tracking**: TorrentPexState maintains the current set of connected peers for each torrent. This is updated on every tick by scanning active peers, and cleaned up via removePeer on disconnect.

3. **Lazy allocation**: PEX state (both per-torrent and per-peer) is allocated on first use, so private torrents and peers that don't support ut_pex incur zero overhead.

4. **Private torrent enforcement**: PEX is completely disabled for private torrents at three levels:
   - Extension handshake omits ut_pex advertisement (existing, BEP 27)
   - Incoming ut_pex messages from peers are silently ignored (existing guard)
   - checkPex skips private torrents entirely (new)

5. **Connection attempts**: PEX-discovered peers are connected through the existing addPeerForTorrent machinery, respecting all connection limits (global, per-torrent, half-open).

6. **BEP 11 limits**: Maximum 50 added and 50 dropped peers per message, 60-second interval between messages per peer.

## What was learned

- Zig's `@ptrCast` from `sockaddr` to `sockaddr.in` requires `@alignCast` because sockaddr has alignment 2 while sockaddr.in has alignment 4.
- The bencode encoder accepts `[]const u8` byte strings directly, which makes building compact peer address blobs straightforward without intermediate allocations.
- Zig's `packed struct(u8)` maps directly to BEP 11 peer flag bytes with correct bit ordering.

## Remaining work

- **IPv6 PEX testing**: The implementation handles IPv6 (added6/dropped6 fields) but has not been tested with real IPv6 peers.
- **Integration testing**: PEX has unit tests but no integration test with multiple daemon instances. The demo_swarm.sh script could be extended to verify PEX peer discovery.
- **PEX flood protection**: BEP 11 recommends rate-limiting connections from PEX to prevent amplification. Currently we rely on half-open limits but don't distinguish PEX-sourced connections for tighter limits.
