# API Authentication (qBittorrent-compatible)

## What was done

Implemented session-based API authentication matching qBittorrent's auth flow:

1. **New `src/rpc/auth.zig` module**: `SessionStore` with fixed-size array of 10 session slots (no heap allocation for session tracking). Each session is a 32-char random hex SID + last-active timestamp. Sessions expire after 1 hour of inactivity. When slots are full, the oldest session is evicted.

2. **Auth endpoints in `src/rpc/handlers.zig`**:
   - `POST /api/v2/auth/login` -- validates `username=X&password=Y` form body, returns `Set-Cookie: SID=<hex>; HttpOnly; path=/`
   - `GET /api/v2/auth/logout` -- invalidates the session from the cookie

3. **Auth enforcement**: All API endpoints except `/api/v2/auth/login` require a valid SID cookie. Returns 403 Forbidden if missing or invalid.

4. **Cookie parsing in `src/rpc/server.zig`**: `parseRequest` now extracts `Cookie: SID=xxx` from HTTP headers into `Request.cookie_sid`. `Response` now supports `extra_headers` for Set-Cookie.

5. **Config in `src/config.zig`**: `api_username` (default "admin") and `api_password` (default "adminadmin") in `[daemon]` section.

6. **`varuna-ctl` updated**: Auto-logins before every command using configured credentials. Supports `--username` and `--password` global flags. Sends `Cookie: SID=xxx` on all subsequent requests.

## Design decisions

- **No heap-allocated session map**: Used a fixed array of 10 slots with LRU eviction. 10 concurrent sessions is sufficient for a headless daemon CLI tool. Avoids needing an allocator in the auth module.
- **`handle()` changed from `*const ApiHandler` to `*ApiHandler`**: The auth handler mutates the session store, so the top-level dispatch needs a mutable pointer. Internal handlers remain `*const` since they only read torrent state.
- **varuna-ctl logins per invocation**: Since ctl creates a new TCP connection per command anyway, it's simplest to login at the start of each invocation. No SID persistence needed.
