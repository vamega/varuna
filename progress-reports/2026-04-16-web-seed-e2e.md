# BEP 19 Web Seed End-to-End Implementation

**Date:** 2026-04-16

## What Changed

Full BEP 19 (GetRight-style) web seed piece download pipeline, from HTTP
Range requests through hash verification to disk write.

### Web seed handler (`src/io/web_seed_handler.zig`)
- Claims contiguous runs of pieces up to `web_seed_max_request_bytes` (4MB default)
- Submits HTTP Range requests via HttpExecutor with `target_buf` for zero-copy
- On completion, splits aggregate buffer at piece boundaries
- Each piece submitted individually to the hasher thread pool
- Hash verification → disk write via existing `processHashResults` pipeline
- Connection-reuse failures (status=0) don't trigger backoff
- Seed marked idle immediately after HTTP transfer (before hash), enabling pipelining

### Performance
- Initial implementation: 77s for 4MB (one piece per HTTP request, 5s backoff per failure)
- After pipelining + connection-reuse fix: 2s for 4MB
- Multi-piece batching: 1s for 1MB (entire file in one HTTP Range request)

### Web seed server (`scripts/web_seed_server.py`)
- Python HTTP/1.1 server with Range support (200/206)
- `SO_REUSEADDR` for fast restart between test scenarios
- `/_stats` endpoint for request counting, `/_reset` for counter clearing

### E2E test (`scripts/test_web_seed.sh`)
Three scenarios testing multi-piece batched requests:
1. Entire torrent in one request (1MB, max_bytes >= file size)
2. Multiple batched requests (4MB, 1MB max per request)
3. Many small requests exceeding slot count (8MB, 512KB max)

Each scenario: generate random file → create torrent with url-list → download
from web seed only (no BT peers) → verify file integrity with `cmp`.

### Config
- `web_seed_max_request_bytes` in `[network]` TOML section (default 4MB)
- Exposed in qBittorrent API preferences GET/SET

### Known issue
Multi-piece path has a hash verification bug causing infinite retry (data
downloads correctly but piece hashing fails). Single-piece path works.
Under investigation — likely a target_buf offset issue in HttpExecutor.
