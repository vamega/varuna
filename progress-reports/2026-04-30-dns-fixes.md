# DNS resolver: connect-failure invalidate + bounded TTL honoring

Closes Open Question #1 from `docs/custom-dns-design.md` â€” two contained
correctness fixes that are independent of the larger custom-DNS-library
implementation question.

## What changed

### Fix 1 â€” connect-failure invalidation
`DnsResolver.invalidate(host)` was exposed but no caller ever invoked
it. When a tracker IP migrated, every subsequent announce kept hitting
the same dead IP for the full cache TTL window.

Wired in via a shared classifier: `dns.shouldInvalidateOnConnectError(err) -> bool`
(`src/io/dns.zig:43`). Consumers:

| Consumer                                | File:line                                    | Trigger                                                 |
|-----------------------------------------|----------------------------------------------|---------------------------------------------------------|
| `HttpExecutor.httpConnectComplete`      | `src/io/http_executor.zig:556`               | io_uring connect CQE error                              |
| Former plain HTTP connect path          | Former synchronous HTTP module               | `posix.connect` error                                   |
| Former HTTPS connect path               | Former synchronous HTTP module               | `posix.connect` error                                   |
| `UdpTrackerExecutor.checkTimeoutsAndRetransmit` | `src/daemon/udp_tracker_executor.zig:660` | Overall deadline OR exhausted backoff schedule (BEP 15) |

`tracker/announce.zig` was a thin wrapper around the former synchronous HTTP client and inherited
its invalidate behavior automatically â€” no separate change.

### Error variant classification

**Invalidate** (the resolved IP looks dead):

  - `ConnectionRefused`
  - `ConnectionTimedOut` â€” covers both kernel `ETIMEDOUT` and the
    io_uring `link_timeout` deadline
  - `NetworkUnreachable`
  - `HostUnreachable`

**Keep cache** (the IP is fine, something else broke):

  - `ConnectionResetByPeer`, `BrokenPipe`, `ConnectionAborted`
    â€” established connection, then died mid-stream
  - `OperationCanceled` â€” varuna initiated the cancel
  - `SubmitFailed`, `SocketCreateFailed` â€” local resource failure
  - `RequestTimedOut` â€” overall request budget; can fire after a
    successful connect
  - HTTP parse errors, 4xx/5xx tracker responses â€” by definition
    the IP was reachable enough to talk back

UDP-specific: BEP 15 has no separate connect surface. The analog of
TCP's `ConnectionTimedOut` is "we sent datagrams and nothing came
back." `UdpTrackerExecutor.checkTimeoutsAndRetransmit` invalidates in
both branches that emit `error.TrackerTimeout` (overall deadline +
exhausted retransmit schedule).

### Fix 2 â€” bounded TTL honoring

Old: every cache entry got a fixed 5-minute lifetime. Tracker announce
intervals are 30-60 min, so steady-state re-announces re-resolved
before nearly every request.

New: shared `dns.TtlBounds` (`src/io/dns.zig:71`) with `floor_s = 30`
and `cap_s = 3600`. Both backends have a public `ttl_bounds` field so
callers / future config wiring can override per-instance.

| Backend                              | TTL source                                                          | Behavior                                              |
|--------------------------------------|---------------------------------------------------------------------|-------------------------------------------------------|
| `dns_threadpool` (`-Ddns=threadpool`, default) | `getaddrinfo` â€” does NOT expose TTL                                   | Fixed cache lifetime = `cap_s` (1 h)                  |
| `dns_cares` (`-Ddns=c_ares`)         | `ares_addrinfo_node.ai_ttl` from `ares_getaddrinfo`                 | `clamp(ai_ttl, floor_s, cap_s)` per record            |

The threadpool default 5 min â†’ 1 h is a ~12Ă— steady-state lookup
reduction. The c-ares variant switched from deprecated
`ares_gethostbyname` to modern `ares_getaddrinfo`, which is the only
path that actually exposes per-record TTL in c-ares's public API.

A TTL â‰¤ 0 in c-ares (e.g. synthetic `/etc/hosts` records) falls back
to the floor so the cache still does something useful.

## Surprises / follow-ups

- **`-Ddns=c_ares` build is broken on `main`** â€” pre-existing, not my
  change. Vendored c-ares (`vendor/c-ares/`) ships only the
  `ares_build.h.cmake`/`.dist`/`.in` template files; `ares_build.h`
  itself isn't generated. Verified by `git checkout main` and
  re-running `nix develop --command zig build -Ddns=c_ares` â€” same
  failure (`'ares_build.h' file not found`). The dns_cares.zig changes
  in this round compile against the same vendored headers and follow
  the same API shapes that the prior code used; full c-ares validation
  must wait for a separate fix to the vendored c-ares build (out of
  scope).
- **`ttl_bounds` is not yet wired into the user-facing config**. The
  field is public on the resolver so the existing `[network]` config
  could plumb it through in a future small change. Not done here to
  keep scope tight.
- **The custom-DNS library round** (`docs/custom-dns-design.md`)
  remains the larger follow-up. These two fixes specifically retire
  the two correctness gaps from that doc's "open questions" section
  that were independent of the library decision.
